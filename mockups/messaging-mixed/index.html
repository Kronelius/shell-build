Mixed-Channel Messaging — three takes

Same data on every screen: Morgan Choi at Lakeside Office Park, with a mixed week of SMS + email across two different email subjects (a proposal thread and an invoice thread). The three options diverge on how the timeline is grouped and how reply-to-thread works — the key interaction you flagged.

All three options use a Gmail-style preview-and-expand pattern for emails: long bodies default-collapse to a 3-line snippet with a "Show full message ▾" toggle. Option 3 also has an expandable "Quoted context" panel inside the inline reply showing the message you're replying to in full — both tested live, click around in any option to try it.

Option 1

Pure Timeline

Flat chronological feed of every message regardless of channel or subject. Bubbles carry a channel chip; emails show their subject inline. Closest to GHL / HubSpot.

Reply UX: hover any bubble → quick Reply button focuses the global compose, auto-switches channel + pre-fills Re: subject for email.
View Option 1
 Option 2

Grouped Threads

Same chronological feed, but consecutive emails on the same subject collapse into thread cards (expandable). SMS stays as individual bubbles. Cleaner for high-volume contacts.

Reply UX: each thread card has its own "Reply to thread" button that opens compose pre-targeted at that subject + channel.
View Option 2
 Option 3

Inline Reply Threads

Flat feed plus inline reply boxes — clicking Reply on any message expands a compose right beneath that bubble. Multiple replies can be drafted in parallel for different threads of work.

Reply UX: reply-in-place on any message; the global compose at the bottom is reserved for fresh, untied messages. Slack-style.
View Option 3
How to compare: open each option, scroll the timeline, click around the Reply affordances, then come back here. The data is identical so the only thing changing is the UX. Mobile is laid out at desktop width here for review — the production build would inherit the existing mobile-pane collapse rules.

Note: these are static mockups — buttons aren't wired. They're for deciding which UX to commit to before doing the schema migration (move channel from Conversation to Message) + component refactor in the live app.
